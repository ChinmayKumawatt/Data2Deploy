DATA VALIDATION FLOW

1. Start `initiate_data_validation()`.
2. Log that the validation stage has started.
3. Read `config.train_path`.
4. Read `config.test_path`.
5. Validate the train file path:
6. Ensure the path is not empty.
7. Ensure the file exists.
8. Ensure the path points to a file.
9. Load train CSV into a pandas DataFrame.
10. Log train dataset shape.
11. Validate the test file path with the same checks.
12. Load test CSV into a pandas DataFrame.
13. Log test dataset shape.
14. Run data integrity checks.
15. Ensure train DataFrame is not empty.
16. Ensure test DataFrame is not empty.
17. Compare train and test column sets.
18. Flag mismatch if train and test columns differ.
19. Store integrity results in the report.
20. Run schema validation.
21. Read expected schema from `config.schema`.
22. Support schema as either:
23. `{column_name: dtype}`
24. or `{"columns": {column_name: dtype}}`
25. Check for expected columns missing from train/test.
26. Check for unexpected columns present in train/test.
27. For expected columns, compare actual pandas dtypes with expected dtype.
28. Record dtype mismatches.
29. Store schema validation results in the report.
30. Run missing value analysis.
31. Read `missing_value_threshold` from config.
32. Compute missing percentage for each train column.
33. Compute missing percentage for each matching test column.
34. Flag columns where missing percentage exceeds the threshold.
35. Store missing value summary in the report.
36. Run target column validation.
37. Read `target_column` from config.
38. Ensure the target column exists in train.
39. Ensure the target column exists in test.
40. Count unique target values in train and test.
41. Raise an error if train target has fewer than 2 unique values.
42. Raise an error if test target has fewer than 2 unique values.
43. Compute class distribution for train and test targets.
44. Read `imbalance_threshold` from config.
45. Add a warning if class imbalance is severe.
46. Store target validation results in the report.
47. Run basic data drift detection.
48. Read `drift_threshold` from config.
49. Exclude the target column from drift checks.
50. Compare only numeric columns present in both train and test.
51. For each numeric column, compute train mean and std.
52. Compute test mean and std.
53. Build a normalized drift score from mean/std differences.
54. Flag the column if the score exceeds the threshold.
55. Store drift results in the report.
56. Combine all statuses:
57. integrity status
58. schema status
59. missing-value status
60. target-column status
61. drift status
62. Set overall status to `PASS` only if all checks pass.
63. Resolve `config.report_path`.
64. Create the parent directory for the report if needed.
65. Save the validation report as JSON.
66. Log the report path and final validation status.
67. Return `DataValidationArtifact`.
68. Artifact includes:
69. `train_file_path`
70. `test_file_path`
71. `report_file_path`
72. `validation_status`
73. If a critical step fails, wrap the error in `CustomException`.

DEBUGGING CHECKPOINTS

1. Confirm `train.csv` and `test.csv` were created by ingestion.
2. Confirm both files are non-empty and readable.
3. Check whether train and test columns match exactly.
4. Check whether the configured schema matches actual column names.
5. Check whether expected dtype strings are compatible with pandas dtypes.
6. Check whether too many missing values are causing validation failure.
7. Check whether the target column exists in both datasets.
8. Check whether the target has at least 2 classes in both train and test.
9. Check whether class imbalance is only a warning or an actual blocker for your pipeline.
10. Check whether numeric columns are being flagged for drift due to threshold settings.
11. Open `validation_report.json` and inspect the failed section first.
